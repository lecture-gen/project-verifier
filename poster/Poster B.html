01

작품 개요

생성형 AI를 쓰면 학생은 코드와 문서를 빠르게 만들어 낼 수 있다. 그래서 제출물만 봐서는 학생이 그 내용을 직접 이해하고 작성했는지, 아니면 상당 부분을 AI나 다른 사람에게 맡겼는지 가려내기 어렵다. 평가자에게는 결과물의 완성도와 별개로 학생이 실제로 수행했는지 확인하는 일이 남는다.

Dialearn은 학생이 올린 프로젝트 자료(ZIP 또는 GitHub 저장소)를 분석하고, 그 자료에 근거한 질문으로 인터뷰한다. 답변이 제출물과 얼마나 맞아떨어지는지 따져, 이 학생이 이 프로젝트를 직접 수행했는지를 영역별 신뢰도와 근거를 담은 리포트로 보여 준다.

02

주요 기능

RAG 자료 분석

학생이 낸 ZIP이나 GitHub 저장소에는 코드 파일과 보고서, 발표자료가 수십 개씩 섞여 있다. Dialearn은 이 자료를 역할별로 분류한 뒤 코드는 구문 트리(AST)로 함수와 클래스 단위, 문서는 제목과 섹션 단위로 나눈다. 이때 소스에 들어 있는 API 키나 비밀번호 같은 민감정보는 자동으로 가린다. 나눈 조각은 벡터로 바꿔 Qdrant에 저장하고, 질문을 만들 때마다 그 프로젝트에서 근거가 될 부분을 찾아 함께 쓴다.

Bloom 유형별 출제

좋은 검증은 학생이 무엇을 만들었는지에서 그치지 않고 왜 그렇게 했고 어떻게 동작하는지까지 묻는다. Dialearn은 Bloom 인지 6단계, 곧 기억, 이해, 적용, 분석, 평가, 창안에 맞춰 질문을 고르게 나누고, 질문마다 답을 채점할 기준을 같이 만든다. 그래서 단순 암기부터 설계 판단까지 학생의 이해를 여러 수준에서 확인한다.

루브릭 기반 꼬리질문

채점은 질문마다 정해 둔 여러 루브릭 기준으로 한다. 답변이 그중 일부 기준을 채우지 못하면 Dialearn은 아직 확인되지 않은 그 기준을 겨냥해 꼬리질문을 던진다. 이미 확인한 내용은 다시 묻지 않고 빠진 부분만 파고들어서, 적은 질문으로도 이해의 빈틈을 메운다.

다중 에이전트 채점과 리포트

컨텍스트 분석, 품질 평가, 질문 생성, 답변 채점, 리포트 작성을 맡은 다섯 개의 AI 에이전트가 차례로 일한다. 마지막 에이전트는 모든 답변을 모아 기술 영역별 신뢰도와 Bloom 단계별 도달 정도를 계산하고, 학생이 어디를 잘 이해했고 어디가 부족한지 강점과 보완점으로 정리해 리포트를 낸다.

03

시스템 구조

서비스 전체를 Oracle Cloud VM 한 대에 Docker Compose로 올린다. 리버스 프록시(nginx)가 HTTPS로 들어온 요청을 Next.js 웹과 FastAPI 백엔드로 넘긴다. 백엔드는 제출 자료를 RAG 파이프라인으로 색인해 Qdrant 벡터 DB에 담고, 평가·세션·리포트는 SQLite에 저장하며, 분석·출제·채점과 음성(STT·TTS)은 OpenAI API로 호출한다.

04

동작 원리

평가자가 자료를 올리면 시스템이 프로젝트 컨텍스트와 품질을 분석하고, 자료를 근거로 Bloom 6단계 질문과 채점 루브릭을 만든다. 학생은 텍스트나 음성으로 답하고, 채점 에이전트가 답변마다 루브릭으로 점수를 매긴다. 근거가 부족하면 꼬리질문으로 더 캐묻는다. 모든 턴이 끝나면 영역별 신뢰도와 도달도를 담은 검증 리포트가 나온다.

06

기대 효과

진위 검증의 자동화

교수자가 학생을 한 명씩 면담하던 일을 시스템이 대신 한다. 출제와 채점이 자동으로 돌아가므로, 평가자는 면담 부담을 덜고 리포트를 검토해 판단하는 데 집중한다.

근거에 기반한 객관적 평가

질문은 학생이 낸 자료에서 나오고, 채점 기준은 학생마다 같다. 평가자에 따라 질문의 깊이나 잣대가 달라지던 문제가 줄고, 여러 학생에게 같은 기준을 적용할 수 있다.

영역별로 들여다보는 이해도

점수 하나로 묶지 않고 백엔드, 프론트엔드, AI처럼 기술 영역을 나눠 영역마다 신뢰도와 이해 수준을 보여 준다. 학생이 프로젝트의 어느 부분을 직접 소화했고 어디가 약한지 드러난다.
